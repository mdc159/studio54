## Real Problems with This Proposal

### 1. Strict Priority Dequeuing Creates an Operational Trap

The proposal accepts P3 starvation during overload and says the response is "add worker capacity." But if the system is under sustained overload, adding workers is exactly what you can't do quickly. The 2-hour alert threshold means P3 items (including digest summaries that users expect) can be silently dropped or delayed for hours before anyone acts. The proposal doesn't address what happens to P3 queue depth during that window — it will grow unboundedly. There's no maximum queue depth, no backpressure, and no TTL on P3 items.

### 2. The DAU Denominator Argument Is Circular

The proposal says events should be divided by DAU because "notifications are generated by social actions, which only active users perform and receive." But this isn't true. A user who hasn't opened the app in a week can still receive notifications because *other* users acted on their content. The "events distributed across opted-in DAU" calculation systematically underestimates per-user event rates for popular users and overestimates rates for low-engagement users. The Poisson model assumes a uniform λ across users, which is almost certainly wrong for a social app where engagement is power-law distributed. The 1.06× reduction factor may be significantly wrong.

### 3. The Bus Factor Mitigation Is Theater

The proposal acknowledges E3 is a bus-factor-one risk on TCPA compliance, then mitigates it with "E1 reviews PRs" and "a quarterly wiki review." PR review doesn't transfer operational knowledge. If E3 is unavailable during a TCPA incident, E1 reviewing PRs historically doesn't mean E1 can diagnose a suppression cache failure under time pressure. The proposal calls this "partially mitigated" but doesn't quantify what "partially" means in terms of actual risk reduction.

### 4. The Suppression Cache TCPA Gap Is Acknowledged But Not Bounded Adequately

The proposal references "an explicit decision about the compliance boundary" and a "bounded durability gap" but Section 2.6 is cut off in this document. The gap is called out in the executive summary as being specified with "probability and operational response," but readers of this proposal cannot evaluate whether those decisions are sound. This is a material omission for a compliance-critical component.

### 5. The Recalculation Job Has No Enforcement Teeth for New Configurations

The proposal says the worker count recommendation requires "human approval" and fires an alert if deviation exceeds 20% for 24 hours. But the 24-hour window means the system can be misconfigured for an entire day before alerting. For a configuration change that could meaningfully affect delivery latency or drop rates, this is a long silent failure window. The "human approval" step has no specified timeout or escalation path if the approval doesn't happen.

### 6. UUID v7 Event IDs Don't Actually Solve the Problem They're Implied to Solve

The proposal mentions UUID v7 (time-ordered) for event IDs but doesn't explain what ordering guarantee this provides in a distributed ingestion scenario. UUID v7 ordering is based on the clock of the issuing node. With multiple ingestion API instances, events from different nodes will have interleaved timestamps with no global ordering guarantee. If any downstream component depends on UUID v7 ordering for correctness (deduplication, sequencing), that assumption is wrong.

### 7. The 0.01% Duplicate Rate Target Has No Statistical Basis

The proposal states the duplicate delivery target is <0.01% of delivered events, but there's no analysis of whether the proposed crash-recovery mechanism can actually achieve this. The measurement mechanism is referenced as being in Section 5, which isn't included in this document. The target is stated as if it's achievable without any analysis of the recovery job's behavior during the failure scenarios that cause duplicates.

### 8. "Drain P0 Before Touching P1" Is Undefined Under Concurrency

The dequeuing pseudocode shows a single-threaded priority check, but the actual system has a worker pool. With N concurrent workers each running this loop independently, there's no coordination. Worker A might be processing P1 while Worker B simultaneously checks and finds P0 empty (because Worker A hasn't finished yet and hasn't enqueued any new P0 items). The "strict priority dequeuing" guarantee described doesn't hold in a concurrent worker pool without a coordination mechanism that isn't specified.

### 9. The Fanout Multiplier of 1.2 Is Underspecified

The proposal uses a fanout multiplier of 1.2 and labels it "direct interactions; excludes viral fanout." For a social app, this is the number that most needs justification. What does 1.2 mean empirically? If the app has any follow/friend graph mechanics, a single post can notify hundreds of followers. The exclusion of viral fanout is a design decision with no specified threshold — at what point does a high-follower user's action get handled differently? If it isn't handled differently, the 18M/day figure could be wrong by an order of magnitude for apps with uneven follower distributions.

### 10. The In-App Channel Is Architecturally Underspecified Relative to Its Volume

In-app is 14.4M events/day — the largest channel by volume — and the proposal says it's "handled via Redis Streams, not sorted sets (see Section 3.4)" which is also not included in this document. The WebSocket delivery path is mentioned as "fully specified" in the executive summary as a correction from a prior version, but that specification isn't present here. This is the highest-volume, lowest-latency channel and it's essentially absent from the visible design.

### 11. The CI Enforcement of Event Type Classification Doesn't Cover Runtime Event Sources

The proposal says the CI check "rejects PRs that add event types without a priority assignment" and the router rejects unclassified events at runtime. But this only covers event types that go through the application's own code. If any external system, partner integration, or internal service calls the Event Ingestion API with an event type not in the classification file, the result is an HTTP 422 and a logged error. There's no specification of how those errors are monitored, who owns the response, or what the SLA is for onboarding a new event type. The "no silent defaulting" guarantee becomes an availability problem if a high-volume caller sends unclassified events.
