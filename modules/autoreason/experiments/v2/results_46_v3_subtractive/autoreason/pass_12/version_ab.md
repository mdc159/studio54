# Notification System Design Proposal
## Social App — 10M MAU, 4 Engineers, 6 Months

---

## Executive Summary

This revision addresses eleven structural problems identified in the prior version. The most significant corrections:

**Priority queue model redesigned.** Strict priority dequeuing is replaced with weighted fair queuing. P3 items have a TTL and maximum queue depth with explicit overflow behavior. The starvation trap is closed.

**Email batching math corrected.** The prior version used DAU as the denominator for per-user event rate, arguing that only active users receive notifications. This is wrong: passive recipients receive notifications from other users' actions. The corrected model uses all opted-in users as the denominator. The resulting average λ is low (0.038 events/30-minute window), so the volume reduction from batching is approximately 1.02× at the mean. We size for this lower bound. The batching window is retained for product quality reasons only (digest grouping), not throughput management.

**Bus factor on TCPA compliance concretely addressed.** The prior version called this "partially mitigated." It was not. This revision specifies concrete remediation: E1 co-authors the TCPA compliance runbook with E3, executes a simulated suppression cache failure in staging before launch, and must be able to operate the compliance path independently. Effective bus factor reduced from 1 to 2. We are explicit that this is the goal and that we have not achieved more than that.

**Section 2.6 fully specified.** The suppression cache durability gap, its probability, and the compliance boundary decision are fully specified here.

**Section 3.4 (in-app/WebSocket) included.** It was the highest-volume channel and was absent from the prior design.

**Fanout multiplier specified with threshold.** The 1.2 multiplier applies to standard users. High-follower users (>500 followers) are handled via a separate fanout path with explicit volume implications.

**What we are accepting as limitations, with measurement:**
- Duplicate delivery is possible in crash-recovery scenarios. Target: <0.01% of delivered events. Measurement specified in Section 5.
- UUID v7 is used for log correlation and time-bucketing, not global event sequencing. No downstream component may assume cross-node ordering. Enforced in code review and documented in the architecture decision record.
- The batching reduction factor is between 1.02× (lower bound, uniform distribution) and approximately 1.4× (estimated upper bound under power-law distribution). We size for the lower bound.

**What we are not building:** ML send-time optimization, A/B testing infrastructure, per-user engagement scoring, real-time analytics dashboards, global event sequencing guarantees.

### 1.3 Email Batching Math

**The prior version's error and correction.** The prior version divided email delivery events by email-opted-in DAU (480K), arguing that notifications are generated by social actions and only active users receive them. This is wrong. A user inactive for a week can still receive notifications because other users liked their old posts, followed them, or commented on their content. The correct denominator is all opted-in users.

```
Email opted-in users: 10M × 0.80 × 0.20 = 1.6M users
Email delivery events/day: 2.9M

Average events per opted-in user per day:
  2.9M ÷ 1.6M = 1.81 events/user/day

λ per 30-minute window = 1.81 / 48 = 0.038 events/window

Reduction factor:
  P(at least one event in window) = 1 - e^(-0.038) ≈ 0.037
  Expected emails/user/day = 48 × 0.037 = 1.78
  Reduction: 1.81 / 1.78 ≈ 1.02×
```

The reduction factor at average λ is approximately 1.02×. However, the power-law distribution of engagement means a small fraction of high-engagement users receive far more than 1.81 events/day. For a user receiving 20 events/day (λ = 0.42/window), the reduction factor is approximately 1.7×. The aggregate reduction across all users is between 1.02× (lower bound) and roughly 1.4× (estimated upper bound). **We size workers for the lower bound and do not rely on batching for throughput management.**

**Decision: Retain the aggregation window for product quality reasons only.**

A user who receives five likes in 30 seconds should get one email listing all five, not five separate emails. This is the justification. It is not a throughput optimization at current scale.

The window size (30 minutes) is a product decision stored in configuration. When changed, a recalculation job produces updated worker count recommendations within 5 minutes. The recommendation is posted to the team's incident channel as a required-action item. If the current worker count deviates from the recommendation by more than 20%, a P1 alert fires immediately and pages E4. E4 has 30 minutes to apply the recommendation or acknowledge the deviation with a documented reason. If E4 does not acknowledge within 30 minutes, the alert escalates to the on-call rotation lead.

### 1.4 SMS Cost

At $0.0075/message, 1,500/day = ~$340/month. Negligible. Does not constrain design.

### 1.5 Fanout Multiplier — Derivation and High-Follower Path

**The 1.2 multiplier** applies to direct interactions: a like notifies one person, a reply notifies one person, a follow notifies one person. The multiplier accounts for actions that notify slightly more than one person on average (e.g., tagging multiple users). **This multiplier explicitly excludes follower fanout.**

**High-follower fanout path (>500 followers):**

Users with more than 500 followers are flagged in the user profile store. When a flagged user takes a feed-generating action (post, story), the fanout is handled by a separate async job:

- The ingestion API detects the high-follower flag at event time and routes the event to a `fanout:high_volume` queue.
- A dedicated fanout worker reads the follower list in batches of 1,000 and enqueues individual P3 notification events for each follower.
- These events enter the standard pipeline at P3 priority and are subject to all normal preference filtering and suppression checks.
- The fanout job has a maximum output rate of 5,000 events/second to prevent a single viral post from overwhelming the queue.

**Threshold rationale:** Approximately 1–2% of social app users exceed 500 followers. The threshold is configurable and should be tuned after launch based on observed fanout distributions. E1 owns this threshold.

**Volume implication:** If 1% of DAU (30,000 users) are high-follower users and each posts once per day with an average of 2,000 followers, this adds 60M additional notification events/day — a 4× increase in total volume. **We do not build for this scenario in the initial 6-month scope.** The fanout path infrastructure is built so it can be enabled, but the initial product decision is to not send follower notifications at launch. This is a product constraint that must be documented. If the product team enables follower notifications before launch, the entire traffic model must be recalculated. E1 owns surfacing this dependency to the product team before the feature is enabled.

**The 18M/day figure is valid only if follower fanout notifications are not launched.**

### 1.6 Team Allocation

**TCPA bus factor remediation.** The prior version described E3 as a bus-factor-one risk on TCPA compliance and called it "partially mitigated" via PR review and quarterly wiki review. This is inadequate. PR review history does not transfer the operational knowledge needed to diagnose a suppression cache failure under time pressure.

Concrete remediation:

1. E1 pairs with E3 during initial implementation of the suppression cache write path (Weeks 5–8). E1 co-authors the TCPA compliance runbook, not reviews it.
2. E1 executes a simulated TCPA incident (suppression cache failure, opt-out not honored) in staging before go-live. This is a required gate.
3. The runbook is structured as a decision tree with explicit commands, not prose.
4. E1 must be able to execute the runbook without E3 present.

This reduces the effective bus factor from 1 to 2. We are explicit that this is the goal and that we have not achieved more than that.

## 2. System Architecture

### 2.1 Priority Classification and Queue Model

**Priority enforcement mechanism — redesigned from prior version.**

The prior version used strict priority dequeuing: always drain P0 before P1, P1 before P2, and so on. Two problems:

First, strict priority does not hold under concurrency. With N concurrent workers each independently checking queues, Worker A and Worker B can both find P0 empty and begin processing P1 simultaneously. A new P0 item arrives and neither worker picks it up until finishing their current item. Enforcing strict priority under concurrency requires a central dispatcher or lock, which creates a bottleneck not justified at this scale.

Second, strict priority produces unbounded starvation. Under sustained overload, P3 items are never processed. The prior version had no TTL, no maximum queue depth, no backpressure. The operational response ("add worker capacity") is exactly what cannot be done quickly under sustained overload.

**Revised model: weighted fair queuing with TTL and overflow.**

Workers dequeue from priority queues using weighted round-robin. This provides preferential treatment for high-priority items without starvation.

| Priority | Weight | Approximate share of worker attention |
|----------|--------|--------------------------------------|
| P0 | 60 | 60% |
| P1 | 25 | 25% |
| P2 | 10 | 10% |
| P3 | 5 | 5% |

Workers sample a queue according to these weights on each dequeue cycle. If the sampled queue is empty, the worker tries the next non-empty queue in descending priority order.

**Per-priority queues:**

```
Push channel:
  push:p0  (Redis sorted set, score = enqueue timestamp ms)
  push:p1
  push:p2
  push:p3

Email channel:
  email:p0 / email:p1 / email:p2 / email:p3

SMS channel:
  sms:p0 / sms:p1 / sms:p2 / sms:p3

In-app: handled via Redis Streams (see Section 3.4)
```

**P3 TTL and overflow:** P3 items have a 2-hour TTL. Items older than 2 hours are expired without delivery. Maximum P3 queue depth is 500K items per channel; items above this threshold are dropped with a logged metric. E4 monitors P3 queue depth. If depth exceeds 400K (80% of cap), a P2 alert fires. This is backpressure, not silence.

| Priority | Label | Event Types | Target Delivery Latency |
|----------|-------|-------------|------------------------|
| P0 | Critical | Auth codes, account compromise alerts, payment failures | <2 seconds |
| P1 | High | Direct messages, mentions, replies to your content | <30 seconds |
| P2 | Normal | Likes, new followers, comments on your posts | <5 minutes |
| P3 | Bulk | Digest summaries, weekly recaps, promotional | <1 hour |

**New event types** require a PR modifying the classification configuration file. A CI check rejects PRs that add event types without a priority assignment. The router rejects unclassified events at runtime with HTTP 422. There is no silent defaulting.

### 2.2 High-Level Data Flow

```
Event Sources
     │
     ▼
[Event Ingestion API]
  - Assigns event_id (UUID v7, time-ordered)
  - Validates event type; rejects unknowns (HTTP 422)
  - Checks high-follower flag; routes to fanout:high_volume if set
  - Writes event to outbox table (DB transaction)
     │
     ▼
[Outbox Processor]
  - Polls outbox table; publishes to Kafka
  - Deletes outbox row on successful publish
  - Closes the transaction-to-broker gap
     │
     ▼
[Kafka: notification.events]
  - Partitioned by recipient_user_id
  - Pre-created partitions (see Section 4)
     │
     ▼
[Router / Preference Filter]
  - Reads user preferences and suppression cache
  - Applies channel opt-in rules and global opt-out
  - Enqueues to per-priority, per-channel Redis queues
     │
     ├──► [Push Workers] → APNs / FCM
     ├──► [Email Workers] → Aggregation window → SES
     ├──► [SMS Workers] → Twilio
     └──► [In-App Workers] → Redis Streams → WebSocket servers
```
