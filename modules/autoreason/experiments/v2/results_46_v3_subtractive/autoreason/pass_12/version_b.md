# Notification System Design Proposal — Revision 4
## Social App — 10M MAU, 4 Engineers, 6 Months

---

## Executive Summary

This revision addresses eleven structural problems identified in Revision 3. The corrections are substantive, not cosmetic. The most significant changes:

**Priority queue model redesigned.** Strict priority dequeuing is replaced with weighted fair queuing. P3 items now have a TTL and a maximum queue depth with explicit overflow behavior. The starvation trap is closed.

**DAU denominator argument retracted.** The Poisson batching model used DAU as the denominator and argued this was correct because only active users generate notifications. This argument is wrong: passive recipients receive notifications from other users' actions. The model is replaced with a conservative upper-bound estimate that acknowledges power-law distribution and does not rely on the false uniformity assumption. The 1.06× reduction factor is retained as a lower bound, not a point estimate.

**Bus factor on TCPA compliance is a real risk, not a mitigated one.** The proposal previously called this "partially mitigated." It is not adequately mitigated. This revision specifies a concrete remediation: the TCPA compliance path is redesigned so that E1 can operate it independently, with a runbook that has been operationally validated, not just written.

**Section 2.6 is now complete.** The suppression cache durability gap, its probability, and the compliance boundary decision are fully specified here, not referenced as existing elsewhere.

**Section 3.4 (in-app/WebSocket) is now included.** It was the highest-volume channel and was absent from the visible design. That is corrected here.

**Fanout multiplier is now specified with a threshold.** The 1.2 multiplier is retained for standard users with a defined cap. High-follower users (>500 followers) are handled via a separate fanout path with explicit volume implications.

**What we are accepting as limitations, with measurement:**
- Duplicate delivery is possible in crash-recovery scenarios. Target: <0.01% of delivered events. The basis for this target and its measurement are specified in Section 5.
- UUID v7 ordering is used for log correlation and time-bucketing, not for global event sequencing. No downstream component may assume cross-node ordering. This constraint is enforced in code review and documented in the architecture decision record.
- The Poisson batching model assumes a simplified distribution. The actual reduction factor is between 1.06× (lower bound, uniform distribution) and approximately 1.4× (estimated upper bound under power-law distribution for active users). We size for the lower bound.

**What we are not building:** ML send-time optimization, A/B testing infrastructure, per-user engagement scoring, real-time analytics dashboards, global event sequencing guarantees.

### 1.3 Email Batching Math — Revised Model

**The prior version's error and retraction.** Revision 3 argued that events should be divided by email-opted-in DAU (480K) because "notifications are generated by social actions, which only active users perform and receive." This argument is wrong. A user who has not opened the app in a week can still receive notifications because other users liked their old posts, followed them, or commented on their content. Passive recipients are a significant portion of notification recipients on any social app. Using DAU as the denominator systematically underestimates per-user event rates for popular users and overestimates rates for low-engagement users. The Poisson model further assumes uniform λ across users, which is false for a social app where engagement follows a power-law distribution.

**Revised approach: conservative upper-bound sizing.**

Rather than assert a precise λ we cannot validate before launch, we bound the problem:

```
Email opted-in users: 10M × 0.80 × 0.20 = 1.6M users
Email delivery events/day: 2.9M

Average events per opted-in user per day:
  2.9M ÷ 1.6M = 1.81 events/user/day

This is the correct denominator: all opted-in users, not just active ones,
because any opted-in user can receive notifications regardless of their
own activity level.

λ per 30-minute window = 1.81 / 48 = 0.038 events/window

Reduction factor:
  P(at least one event in window) = 1 - e^(-0.038) ≈ 0.037
  Expected emails/user/day = 48 × 0.037 = 1.78
  Reduction: 1.81 / 1.78 ≈ 1.02×
```

**The reduction factor at average λ is approximately 1.02× — even smaller than the prior version's 1.06×.** However, this is the average across all opted-in users. The power-law distribution means a small fraction of high-engagement users receive many more than 1.81 events/day. For a user receiving 20 events/day (λ = 0.42/window), the reduction factor is 1 - e^(-0.42) ≈ 0.34 per window, yielding a per-user reduction of approximately 1.7×. The aggregate reduction across all users is between 1.02× (lower bound, treating all users as average) and roughly 1.4× (estimated upper bound, accounting for the high-engagement tail).

**We size workers and downstream infrastructure for the lower bound (1.02×, essentially no reduction).** We do not rely on batching for throughput management.

**Decision: Retain the aggregation window for product quality reasons only.**

The aggregation window is justified by one reason: digest grouping. A user who receives five likes in 30 seconds should get one email listing all five, not five separate emails. This is a product requirement. It is not a throughput optimization at current scale.

The window size (30 minutes) is a product decision stored in configuration. When changed, a recalculation job produces updated worker count recommendations. The enforcement mechanism for this job is specified in Section 1.3.1.

#### 1.3.1 Configuration Recalculation Job — Enforcement Mechanism

When the batching window parameter changes:

1. The recalculation job runs automatically and outputs revised worker count recommendations within 5 minutes of the configuration change.
2. The recommendation is posted to the team's incident channel as a required-action item.
3. If the current worker count deviates from the recommendation by more than 20%, an alert fires immediately — not after 24 hours. The 24-hour window in Revision 3 was too long; a configuration change takes effect immediately and can cause delivery failures within minutes.
4. The alert is P1 severity. It pages E4. E4 has 30 minutes to either apply the recommendation or acknowledge the deviation with a documented reason.
5. If E4 does not acknowledge within 30 minutes, the alert escalates to the on-call rotation lead.
6. The "human approval" step has a hard 30-minute SLA. There is no open-ended waiting.

### 1.5 Fanout Multiplier — Derivation and High-Follower Path

**The 1.2 multiplier.** For a social app where most interactions are direct (likes on posts, replies to comments, follows), the average number of notification recipients per action is close to 1. A like notifies one person. A reply notifies one person. A follow notifies one person. A post notifies zero people directly (followers are notified separately, see below). The 1.2 multiplier accounts for actions that notify slightly more than one person on average (e.g., tagging multiple users).

**This multiplier explicitly excludes follower fanout.** When a user with followers posts content, we do not notify all followers in the standard pipeline. The reason: a user with 10,000 followers posting once generates 10,000 notification events, which would push the average fanout multiplier from 1.2 to something much larger and would make the 18M/day figure wrong by an order of magnitude.

**High-follower fanout path (>500 followers):**

Users with more than 500 followers are flagged in the user profile store. When a flagged user takes a feed-generating action (post, story), the fanout is handled by a separate async job, not the standard notification pipeline:

- The ingestion API detects the high-follower flag at event time.
- The event is routed to a `fanout:high_volume` queue instead of the standard pipeline.
- A dedicated fanout worker reads the user's follower list in batches of 1,000 and enqueues individual P3 notification events for each follower.
- These events enter the standard pipeline at P3 priority and are subject to all normal preference filtering and suppression checks.
- The fanout job has a maximum output rate of 5,000 events/second to prevent a single viral post from overwhelming the queue.

**Threshold rationale:** 500 followers is a conservative threshold. Approximately 1-2% of social app users exceed this threshold. The exact threshold is configurable and should be tuned after launch based on observed fanout distributions. E1 owns this threshold. It is stored in the classification configuration file alongside event type priorities.

**Volume implication of high-follower fanout:** If 1% of DAU (30,000 users) are high-follower users and each posts once per day with an average of 2,000 followers, this adds 30,000 × 2,000 = 60M additional notification events/day. This is a 4× increase in total volume. We do not build for this scenario in the initial 6-month scope. We build the fanout path infrastructure so it can be enabled, but the initial product decision is to not send follower notifications for the first launch. This is a product constraint, not an engineering one, and it must be documented as a launch limitation.

**The 18M/day figure is valid only if follower fanout notifications are not launched.** If the product team decides to enable follower notifications before or at launch, the entire traffic model must be recalculated and the infrastructure plan revised. E1 owns surfacing this dependency to the product team before the feature is enabled.

### 1.6 Team Allocation

**Resolved ownership conflicts from prior version. Bus factor on TCPA compliance is now concretely addressed.**

The prior version described E3 as a bus-factor-one risk on TCPA compliance and called it "partially mitigated" via PR review and a quarterly wiki review. This is inadequate. PR review history does not transfer the operational knowledge needed to diagnose a suppression cache failure under time pressure. This revision specifies a concrete remediation.

**TCPA bus factor remediation:**

The suppression cache write path and TCPA compliance logic are redesigned so that E1 can operate them independently of E3. Specifically:

1. E1 pairs with E3 during the initial implementation of the suppression cache write path (Weeks 5–8). E1 writes the TCPA compliance runbook jointly with E3, not as a reviewer.
2. E1 executes a simulated TCPA incident (suppression cache failure, opt-out not honored) in a staging environment before the system goes live. This is a required gate, not optional.
3. The runbook is structured as a decision tree with explicit commands, not prose. It is reviewed by E1 and E3 together quarterly, but more importantly, E1 must be able to execute it without E3 present.
4. The compliance boundary (what happens during a cache failure) is specified in Section 2.6, which is fully written in this document.

This does not eliminate the bus factor. It means E1 has operational knowledge of the compliance path, reducing the effective bus factor from 1 to 2. We are explicit that this is the goal and that we have not achieved more than that.

## 2. System Architecture

### 2.1 Priority Classification and Queue Model

**Priority enforcement mechanism — redesigned from prior version.**

**Problem with strict priority dequeuing (Revision 3).** The prior version used strict priority dequeuing: always drain P0 before P1, P1 before P2, P2 before P3. Two problems with this:

First, the concurrency problem: with N concurrent workers each independently checking queues, "drain P0 before touching P1" has no meaning. Worker A checks P0, finds it empty, begins processing P1. Simultaneously, Worker B checks P0, also finds it empty, also begins processing P1. A new P0 item arrives. Neither worker will pick it up until they finish their current P1 item. The "strict priority" guarantee does not hold under concurrency without a coordination mechanism. Adding coordination (a central dispatcher, a lock) creates a bottleneck that is not justified at this scale.

Second, the starvation trap: under sustained overload, P3 items are never processed. Queue depth grows unboundedly. There is no TTL, no maximum depth, no backpressure. The proposal said "add worker capacity" as the response, but under sustained overload, adding capacity quickly is exactly what cannot be done. P3 items (digest summaries) would accumulate silently for hours before the alert fires, at which point the queue may contain millions of stale items.

**Revised model: weighted fair queuing with TTL and overflow.**

Workers dequeue from priority queues using a weighted round-robin with priority-weighted probabilities. This provides preferential treatment for high-priority items without starvation guarantees that cannot be honored under concurrency.

**Dequeuing weights:**

| Priority | Weight | Approximate share of worker attention |
|----------|--------|--------------------------------------|
| P0 | 60 | 60% |
| P1 | 25 | 25% |
| P2 | 10 | 10% |
| P3 | 5 | 5% |
