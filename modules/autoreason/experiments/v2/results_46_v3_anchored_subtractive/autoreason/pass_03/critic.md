## Real Problems Found

### 1. The `user_id % 100` Grouping Breaks Per-User Ordering Claims

The proposal states the scheme gives "weak per-user ordering within a group" and that notifications for the same user "tend to land in the same group." This is accurate, but the proposal also says this is acceptable because strict ordering isn't needed. The problem is that 100 groups means each group contains ~30,000 users at 3M DAU. A single slow message from any of those 30,000 users stalls processing for all other users in that group. The proposal never addresses head-of-line blocking within a group, which is a real throughput problem at this scale, not a theoretical one.

### 2. The Two-Layer Deduplication Has a Race Condition

The router sets `dedup:{dedup_key}` in Redis and then enqueues to SQS. These are two separate operations with no atomicity guarantee. Two concurrent events for the same user/type/entity arriving within milliseconds can both pass the Redis check before either sets the key, resulting in duplicate enqueues. The proposal describes this as catching "duplicate events from upstream" but the design does not actually prevent them.

### 3. In-App Write Is Called "Synchronous" Without Acknowledging What That Means

For P0 security alerts, the proposal says the in-app store write happens "synchronously, before anything else." At 50M notifications/day with P0 events included, a synchronous PostgreSQL write in the hot path of the router means router latency is now coupled directly to database latency. A PostgreSQL slowdown or lock contention event delays all P0 routing. The proposal presents this as a reliability improvement while introducing a new availability dependency.

### 4. The `sent:{notification_id}` Dedup Key Has No Defined Source for `notification_id`

The worker layer checks `sent:{notification_id}` before dispatch. The proposal never defines where `notification_id` comes from, how it is generated, or whether it is stable across SQS redeliveries. If `notification_id` is generated by the worker at dequeue time, it is different on each redelivery and the dedup check provides no protection. The correctness of the entire at-least-once protection depends on a detail that is never specified.

### 5. The FCM Adaptive Rate Limiter Has a Shared Mutable State Problem

`_consecutive_429s` and `_rate_limiter` are instance-level state modified from concurrent async tasks without locks. `reduce_rate` and `_consecutive_429s` increments can race. The proposal presents this code as a solution to the undocumented rate limit problem, but the implementation has a concurrency bug that could cause the rate to be reduced multiple times simultaneously during a burst of 429s, driving throughput to the floor.

### 6. The Empirical Rate Limit Discovery Process Runs in Production

The proposal specifies ramping FCM concurrency from 50 to 500 in production during the first week, in steps, monitoring for 429s. This means real user notifications are being used to probe Firebase's undocumented limits. If the actual limit is lower than 300 req/sec, real P0 security alerts are delayed or dropped during the discovery ramp. This is presented as a responsible engineering practice, not as what it is: using production traffic as a load test against an external dependency.

### 7. Cross-Training Doesn't Survive the Scenario It's Designed For

E1 is cross-trained on push so E2's absence doesn't halt that workstream. But E1's primary responsibility is "core pipeline, SQS queue infrastructure, delivery workers" — the foundational layer everything else depends on. If E2 leaves and E1 shifts to push integrations, the core pipeline loses its primary owner at the same time the team is already under stress. The cross-training addresses the channel integration gap while creating a worse gap in the layer below it.

### 8. The OTP Service Redis Key Has No Cleanup Path for Abandoned Deliveries

The OTP flow writes `otp_delivery:{delivery_id}` with a 300-second TTL. If the Twilio or APNs call hangs indefinitely rather than failing, the key expires as "pending" with no record of whether delivery occurred. The next retry creates a new `delivery_id` and sends a second OTP. The proposal claims the `delivery_id` prevents duplicate delivery, but it only prevents duplicate delivery of the *same* `delivery_id` — a hung first attempt followed by a retry is two separate OTPs in flight simultaneously, which is the exact problem the design claims to solve.

### 9. The P0 Email Escalation Scope Is Undefined

"For the most critical security alerts (account takeover indicators), the router also enqueues an email." The proposal never defines what constitutes an "account takeover indicator" versus a regular P0 security alert. This is a product decision encoded as an engineering implementation detail with no specification. The router has to make this classification in code, but there is no source of truth for what the classification rules are.

### 10. The APNs Token Manager Code Is Cut Off

The `APNsTokenManager.get_token` method is truncated mid-implementation. The lock is acquired on the slow path, but the code inside the lock is never shown. Whether the implementation correctly double-checks token freshness inside the lock (preventing multiple simultaneous refreshes) cannot be evaluated. A proposal in Revision 3 that describes itself as addressing correctness gaps should not have incomplete code in a security-critical component.
